# The Vault Research Lab

Local-first research lab alpha with an Electron cockpit and a Python FastAPI core.

## Quick Start

```bash
pnpm install
cd services/core && uv sync
cd ../..
pnpm dev
```

The desktop app starts the FastAPI core with a random local token, waits for `/health`, then opens the React renderer. For browser-only UI development, run `pnpm --filter @vault/desktop dev:renderer` and `cd services/core && uv run python -m vault_core.main --dev`.

## Useful Commands

```bash
uv run pytest services/core/tests
pnpm test
pnpm e2e
pnpm verify
pnpm verify:strict
./scripts/reset_dev_db.sh
./scripts/generate_contracts.sh
./scripts/validate_ai_registries.sh
./scripts/plan_ai_registry_release.sh
./scripts/hydrate_ai_registry_metadata.sh --model-registry candidate-model-registry.json --output candidate-model-registry.hydrated.json
./scripts/probe_ai_registry_artifacts.sh
./scripts/verify_ai_registry_artifacts.sh --model-registry candidate-model-registry.hydrated.json --runtime-registry candidate-runtime-registry.json --evidence-output candidate-ai-byte-evidence.json
./scripts/prepare_ai_registry_release_candidate.sh --probe-sources --verify-bytes
./scripts/pin_ai_registries.sh
./scripts/check_ai_readiness.sh
./scripts/check_ai_readiness.sh --allow-demo
./scripts/check_ai_readiness.sh --format markdown --output docs/reports/local-ai-readiness.md
./scripts/test_local_ai.sh
./scripts/test_voice_local.sh
```

`pnpm verify` runs the runnable v1 quality gate in demo-allowed mode: Python core tests, Python lint, desktop tests, production build, renderer e2e smoke, AI registry validation, local AI smoke, local voice smoke, and the local-AI readiness check with `--allow-demo`. `pnpm verify:strict` runs the same gate with strict production readiness; it is expected to fail until real production model/runtime manifests, approval evidence, checksums, and capability routes are pinned.

The same runnable v1 gate is wired into GitHub Actions at `.github/workflows/verify.yml` for pushes to `main` and pull requests. The workflow can also be run manually with `strict_production=true` to execute `pnpm verify:strict` for release approval once production model and runtime manifests are ready.

`./scripts/validate_ai_registries.sh` is the structural registry gate: it fails on malformed model/runtime manifests, duplicate IDs, broken pack references, invalid checksum formats, malformed Hugging Face revisions, unsafe artifact or fixture paths, invalid source URLs, unsafe runtime archive members, missing local fixture files, and registry digests that do not match the bundled app-pinned `registry_policy.json`. The same validation contract is exposed at `/ai/registry/validation` and in Settings -> AI Models. Placeholders remain warnings there and hard release blockers in the readiness gate.

`./scripts/plan_ai_registry_release.sh` evaluates bundled or candidate model/runtime registries before pinning. It runs structural validation, summarizes production pack/model/runtime readiness, verifies that approval evidence is present, checks model kind/runtime/capability/default suitability plus pack capability/profile fit, previews the exact manifest IDs and SHA-256 digests that would change, and exits zero only when the candidate manifest set is ready for `pin_ai_registries.sh`. The bundled-registry version is also exposed at `/ai/registry/release-plan` and in Settings -> AI Models as the Registry release plan panel, with `/ai/registry/release-plan/export` and an Export Markdown action for saving a review artifact from the app. Settings can also dry-run selected candidate model/runtime JSON files through `/ai/registry/release-plan/evaluate`, show their pin-readiness ratios and pin-impact preview next to the bundled blockers, export `candidate-ai-registry-release-plan.md` with the selected filenames and digest preview embedded in the audit artifact, probe candidate artifact sources plus license URLs or bundled license paths through `/ai/registry/artifact-probe/evaluate`, export `candidate-ai-registry-artifact-probe.md`, export `candidate-local-ai-approval-template.md` and `candidate-local-ai-evidence-template.json` through `/ai/readiness/approval-template/evaluate` for the same candidate files, and apply the filled reviewer evidence overlay JSON through `/ai/registry/evidence/apply`. The overlay returns patched candidate registries plus exact patched-file SHA-256 digests, validation, release-plan, approval-template, and pin-handoff reports in `candidate-ai-registry-evidence-bundle.json`; Settings can also show the source-probe, release-packet, and final pin handoff commands inline, save the applied release plan Markdown, applied approval checklist Markdown, final pin handoff Markdown, and patched model/runtime registry JSON files separately for the final probe/packet/plan/pin commands. Use `--model-registry ... --runtime-registry ... --format markdown --output ...` to produce the same auditable release-plan artifact from CLI.

`./scripts/hydrate_ai_registry_metadata.sh --model-registry ... --output ...` resolves candidate Hugging Face model entries into immutable commit revisions, exact file size, LFS SHA-256, and upstream license label metadata before reviewer approval. The same hydrator is exposed at `/ai/registry/metadata/hydrate` and in Settings -> AI Models after candidate evaluation, where it can refresh the candidate release plan with the hydrated model manifest digest. It only edits model registry metadata, does not set `approval.*`, does not invent license artifacts, and keeps the approval-template, probe, evidence-overlay, and pin gates authoritative.

`./scripts/probe_ai_registry_artifacts.sh --model-registry ... --runtime-registry ... --format markdown --output ...` probes candidate production artifact source URLs and license references without downloading full model/runtime binaries. It verifies source reachability, remote `Content-Length` against pinned `size_bytes`, remote SHA-256 metadata when the host exposes checksum headers, remote license URL reachability, and bundled `license_path` existence, supports URL and Hugging Face sources, and exits zero only when every candidate source/license probe passes. Full download verification remains the final checksum gate when a source does not expose remote SHA-256 metadata.

`./scripts/verify_ai_registry_artifacts.sh --model-registry ... --runtime-registry ... --evidence-output ...` performs that final candidate byte gate before approval: it streams production model/runtime artifacts from pinned URL or Hugging Face sources, enforces a configurable `--max-bytes` limit, computes exact `size_bytes` and SHA-256 values, blocks pinned size/checksum mismatches, and emits an evidence overlay JSON for `apply_ai_registry_evidence.sh`. It does not install artifacts, does not write into app model/runtime storage, and never sets `approval.*`; reviewers still have to approve and apply the resulting evidence.

`./scripts/prepare_ai_registry_release_candidate.sh --model-registry ... --runtime-registry ... --evidence ... --output-dir ... --probe-sources --verify-bytes` is the one-shot pre-pin packet builder. It applies reviewer evidence, writes patched model/runtime registries, the evidence bundle, applied release plan, applied approval checklist, source-probe report, byte-verification report, byte-evidence JSON, pin handoff, acceptance report, packet index, and packet summary JSON into one directory. Omit `--probe-sources` only when a deterministic offline packet is needed; omit `--verify-bytes` only when release owners intentionally cannot stream candidate binaries in the packet environment. With source probing and byte verification enabled, it exits zero only when the patched candidate manifests are ready to pin, every candidate source/license URL or bundled license-path probe passes, and every patched candidate artifact byte stream matches the pinned size and SHA-256.

`./scripts/pin_ai_registries.sh` rewrites the app-pinned registry policy after an approved model or runtime manifest change. Use `--check --format json` to inspect the current model/runtime registry hashes without modifying the policy file. Use `--model-registry candidate-model-registry.patched.json --runtime-registry candidate-runtime-registry.patched.json --check` to dry-run an approved candidate; add `--format markdown --output candidate-ai-registry-acceptance.md` to save the acceptance report reviewers can attach before promotion. Without `--check`, the command refuses blocked candidates, copies the candidate manifests into the bundled registry files, and writes `registry_policy.json` from the exact copied bytes.

`./scripts/check_ai_readiness.sh` is the strict local-AI release gate: it exits non-zero until production Tiny/Standard/Strong model packs, managed runtimes, privacy boundaries, and capability routes are ready. Capability routes must stay local-only, avoid cloud and mock providers, point at installed app-approved model inventory, avoid fixture/manual-import assets, and pass local runtime testing where applicable. The command also prints an approval board that groups repeated blockers into release-work items, and can export the same contract as a Markdown approval checklist for release review. The same strict-production Markdown dossier is available from `/ai/readiness/report/export` and Settings -> AI Models through Export readiness. `/ai/readiness/approval-template/export` and Settings Export template produce both a fill-in Markdown manifest approval template and a fillable evidence overlay JSON template for source, checksum, size, license, model runtime defaults, and `approval.*` evidence fields; `/ai/readiness/approval-template/evaluate`, Settings candidate template/evidence exports, and `./scripts/export_ai_approval_template.sh --model-registry ... --runtime-registry ... --evidence-output ...` do the same for pre-pin candidate manifests. `./scripts/probe_ai_registry_artifacts.sh` verifies candidate source/license reachability, bundled license-path existence, size headers, and remote checksum metadata when exposed before release approval; `./scripts/verify_ai_registry_artifacts.sh` streams the candidate bytes and emits checksum/size evidence without installing or approving them; `./scripts/apply_ai_registry_evidence.sh --model-registry ... --runtime-registry ... --evidence ... --release-plan-output ... --approval-template-output ... --pin-handoff-output ... --model-output ... --runtime-output ... --check` applies a structured evidence overlay, writes applied Markdown review artifacts, a final pin handoff, and patched registry JSON files when requested, and exits zero only when the patched candidate manifests become ready to pin. Use the packet builder with `--probe-sources --verify-bytes` when reviewers need the full source, byte, evidence, acceptance, and handoff artifact set in a single directory. `--allow-demo` is for dev builds that intentionally permit the fixture pack and demo runtime.

Managed runtime URL sources can point either directly at an executable or at a pinned archive. For archive-backed runtimes, set `source.archive.member` or `source.archive_member` to the exact binary member and optionally set `source.archive.format` or `source.archive_format` to `zip`, `tar`, `tar.gz`, or `tgz`. The installer verifies the downloaded archive against `files[0].sha256` and `files[0].size_bytes`, rejects unsafe member paths, symlinks, and links, extracts only that explicit member into app-managed runtime storage, makes it executable when requested, stores the installed binary checksum, and then runs the normal bounded smoke/version check.

`./scripts/test_local_ai.sh` and `./scripts/test_voice_local.sh` are in-process smoke checks for the local AI and voice subsystems. They default to temporary Vault data directories, exercise the same public API routes as the desktop app, and support `--data-dir` for checking a configured workspace.

## V1 Scope

This alpha implements the product spine:

- durable SQLite workspace, event log, source blocks, review queue, graph, tools, and learning items,
- notes as editable, versioned sources with FTS indexing,
- deterministic local embedding index with vector/hybrid search over source blocks,
- text, Markdown, and text-based PDF import,
- deterministic extraction proposals with quote validation,
- review-gated claim/concept promotion with evidence links,
- scoped grounded assistant answers from indexed evidence,
- manual Night Lab brief generation,
- controlled Python Tool Studio Lite with a built-in Claim Citation Checker,
- secure Electron shell with context isolation, sandboxing, and whitelisted IPC.

Bundled production local model execution, remote AI providers, MCP, cloud sync, autonomous crawler, and packaging are intentionally not enabled in this first runnable product.
